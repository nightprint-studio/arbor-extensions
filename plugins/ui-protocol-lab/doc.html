UI Protocol Lab

A test bench for the plugin UI dispatch + event/patch protocol. Everything is in one modal so you can poke each piece and watch it react live. Open it from the Command Palette (Ctrl+K) → UI Lab: open playground.

Editor widget (CodeMirror 6)

  • Value-bearing — the editor's text is the form field doc; it is submitted like any field and the host pushes content into it with set_value (try Load sample / Normalize / Clear).
  • on_edit — debounced inside the widget (250 ms here) and scoped: the handler gets { node_id, slot = "edit", value, state }, never the whole form. Type and watch the telemetry update.
  • on_select — fires on cursor moves / range selects with { from, to, text }. Click around or select text and watch the Selection line.
  • scope_state — the editor declares { "edits" }, so the slice state.edits rides along in each on_edit payload. The plugin logs the value it sees to prove the round-trip (Settings → Plugins → logs).

Granular updates (no re-mount)

  • patch · set — the Characters / Lines / on_edit fires labels are updated by a deep assign at a path inside each node.
  • patch · merge — the JSON-validity banner shallow-merges variant + text as you type valid / invalid JSON.
  • patch · append / remove — Append note pushes a child into a container; Remove last note splices it back out by id.
  • set_state_path — the edit counter and last selection are written into the form's opaque state one slice at a time.

Command invocation

  • From a button — Apply via command carries dispatch = { kind = "command", id = "ui-protocol-lab::apply", args = { source = "button" } }. It fires the registered apply command instead of a plain action.
  • At runtime — the palette entry UI Lab: fire apply via command.fire calls arbor.command.fire("ui-protocol-lab::apply", …). Cross-plugin invocation works identically: any plugin holding the command_invoke permission can fire an invocable command, gated by the command's declared required tier.
  • Host built-ins — the Host commands (Fase 2b) section fires arbor:repo.refresh and arbor:app.open_settings (run by Arbor itself), both from a button dispatch and from arbor.command.fire("arbor:repo.refresh"). Same gating; these two need no permission tier, while the arbor:git.* built-ins additionally require git = "write".

Dispatch union

Buttons mix both forms on purpose: Load sample / Normalize / Clear use a bare action string (whole-form payload), while Apply via command uses a dispatch object. The select's actions.change is a dispatch target too, so changing it sends a scoped payload rather than the whole form.

Main-area view (add_view)

The same protocol also drives a body surface, not just modals. This plugin registers a view with arbor.ui.add_view — open it from the UI Lab Dashboard icon in the left activity bar, the Command Palette (Open View: UI Lab Dashboard), or Alt+Shift+V.

  • Same renderer — the view body is the full FormNodeRenderer, so sections, editor, buttons and the dispatch / scoped / patch protocol all work exactly as in the modal.
  • on_view_open / on_view_close — the plugin fills the body from on_view_open with arbor.ui.set_panel_content and logs the close.
  • Live patch in the body — Tick updates a label + banner via a single arbor.ui.form.patch (no re-mount); the in-body editor fires a scoped on_edit just like the modal. View node ids are distinct from the modal's, so the shared patch channel never cross-talks.
  • Placement — registered as placement = "graph" (keeps the tab bar + bottom panel); switch it to "main" in main.lua to take over the whole body.

This plugin is a development aid — it touches no repository state. Remove it from index.json when you're done testing.
